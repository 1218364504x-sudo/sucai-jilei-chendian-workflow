# 飞书多维表格设计

## 定位

飞书多维表格只做一件事：快速评级。

不要把它设计成完整资料库。完整资料保存在 Obsidian，精品故事沉淀到飞书知识库。

## 日常视图核心字段

日常评级视图只需要 4 列：

| 字段 | 用途 |
|---|---|
| 标题 | 快速判断故事主题 |
| 故事简介 | 读完整故事梗概 |
| 拍摄可行性 | 判断当前阶段好不好拍 |
| 星级评级 | 用户最终打星 |

## 后台字段

后台可以保留同步所需字段，但不放在日常评级视图里：

- 文件路径
- 分类
- 国家地区
- 来源类型
- 来源链接
- 适合形式
- 开发优先级
- 低成本切口
- AI建议星级
- AI建议理由
- 知识库收纳状态
- 最后修改时间

## 推荐视图

| 视图 | 筛选规则 | 用途 |
|---|---|---|
| 01_待完整判断 | 星级评级为空，或拍摄可行性为空 | 日常主入口，同时补两项 |
| 02_优先开发 | 开发优先级 = 优先开发 | 高星且 A/B，近期可开发 |
| 03_高价值储备 | 开发优先级 = 高价值储备 | 高星但 C/D，长期储备 |
| 04_低成本练手 | 开发优先级 = 低成本练手 | 3 星且 A，适合练手 |
| 05_暂存 | 开发优先级 = 暂存 | 暂不开发也不删除 |
| 06_删除候选 | 开发优先级 = 删除候选 | 每周删除复核 |
| 99_后台全量_同步排查 | 全量 | 排查同步问题 |

各视图建议显示字段：

| 视图 | 建议显示字段 |
|---|---|
| 01_待完整判断 | 标题、故事简介、拍摄可行性、星级评级、AI建议星级、分类 |
| 02_优先开发 | 标题、故事简介、拍摄可行性、开发优先级、星级评级、分类、低成本切口 |
| 03_高价值储备 | 标题、故事简介、拍摄可行性、开发优先级、星级评级、分类、低成本切口 |
| 04_低成本练手 | 标题、故事简介、拍摄可行性、开发优先级、星级评级、分类、低成本切口 |
| 05_暂存 | 标题、故事简介、拍摄可行性、开发优先级、星级评级、分类 |
| 06_删除候选 | 标题、故事简介、拍摄可行性、开发优先级、星级评级、分类、评分理由 |
| 99_后台全量_同步排查 | 全量字段 |

## 自动归类逻辑

用户在飞书填写 `星级评级` 和 `拍摄可行性` 后：

1. 任意一项为空，当前记录留在 `01_待完整判断`。
2. `开发优先级` 暂为 `待完整判断`。
3. 两项都填完后，系统生成最终开发优先级。
4. 星级同步回 Obsidian 的 `星级` 字段。
5. 记录进入优先开发、高价值储备、低成本练手、暂存或删除候选视图。
6. 如果是 4-6 星且拍摄可行性已填，进入精选区和知识库。
7. 自检报告更新统计。

飞书知识库只沉淀两项完整的 4-6 星故事。知识库正文建议按日期拆页，每条故事至少保留：标题、星级、分类、拍摄可行性、开发优先级、低成本切口和故事简介。

## 最容易出错的点

### 1. 用错主键

必须使用 `文件路径` 作为同步唯一标识。不要用标题。

### 2. 待完整判断数量不对

待完整判断不是 `AI建议星级` 为空，而是正式 `星级评级` 或 `拍摄可行性` 任意一项为空。

### 3. 日常视图字段太多

字段越多，评级越慢。飞书表格不是资料库，只保留判断所需信息。

保留 `拍摄可行性` 是为了避免把高价值但难拍的故事误打低分。星级只评故事价值，拍摄难度单独看。

辅助视图里保留 `开发优先级`，是为了让开发、储备、练手、暂存和删除候选都能看到同一套判断。

### 4. 同时在 Obsidian 和飞书改同一字段

需要明确冲突策略。推荐默认 Obsidian 优先，飞书只负责 `星级评级`。
